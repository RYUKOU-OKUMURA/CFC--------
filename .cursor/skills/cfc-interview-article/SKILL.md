# CFC インタビュー記事生成

## 概要

インタビューの文字起こしを入力として、日本キャッシュフローコーチ協会のメディアに掲載するインタビュー記事を構成・執筆するスキル。

**インタビュアー（固定）：** 奥村 龍晃（日本キャッシュフローコーチ協会会員／キャッシュフローコーチ養成塾東京9期）

**想定読者：** キャッシュフローコーチ協会の会員・養成塾生（コンサルタント・士業）

## 入力要件

記事生成に以下の情報が必要。不足している場合はユーザーに確認する。

| 項目 | 必須 | 説明 |
|------|------|------|
| 文字起こし | 必須 | インタビュー録音の書き起こしテキスト |
| 実践者プロフィール | 必須 | 氏名、養成塾の期（例：大阪3期）、肩書、所属 |
| 経歴 | 必須 | 1文にまとめられる職歴・背景 |
| テーマ | 必須 | 何についてのインタビューか（文字起こしから推定可） |

## 記事の全体構造

記事は **3つのブロック** で構成する。

### ブロックA：導入部

```markdown
# タイトル（主題を一文で）
## サブタイトル（テーマの補足）

**実践者：** 氏名（養成塾○○期）肩書

経歴要約（1文）

**インタビュアー：** 奥村 龍晃（日本キャッシュフローコーチ協会会員／キャッシュフローコーチ養成塾東京9期）

リード文第1段落（記事の趣旨を説明）

リード文第2段落（読者のペイン ↔ 実践者の解決策を対比し、記事の読み方を示す）

---
```

**タイトル設計：**

H1とH2はセットで機能させる。H1で「なぜ読むべきか」を示し、H2で「何が起きたか」を示す。

- **H1（読者を引き込む）：** 読者が気づいていない機会・意外な事実・市場の空白を提示する。読者の痛みを直接書くのではなく、「え、そうなの？」という好奇心を引き出す。動詞原形で終わる説明的な一文にしない
  - 例：`企業研修マーケットは、財務の席だけが空いている`
- **H2（具体的な結果を見せる）：** ビフォー→アフターの落差、または具体的な数字・成果を短く示す。H1の好奇心に対して「それで、どうなったの？」に答える役割
  - 例：`未経験・実績ゼロから、研修会社5社と取引するまで`

**H1で避けるべきこと：**
- 養成塾や協会への批判に聞こえる表現（例：「養成塾を出たのに仕事がない」）
- 読者のプライドを傷つける表現（例：「顧問契約が取れない」をH1に置く）
- 汎用的すぎて誰にでも当てはまるコピー

**リード文の設計：**
- 第1段落：「今回は〜を伺うインタビューです。」の形式
- 第2段落：読者が感じるハードル → 実践者のアプローチ → 「その考え方と具体的な進め方を対話形式でまとめます。」で締める

### ブロックB：対話本文

各セクションは以下のパターンで構成する。

```markdown
---

## セクション見出し（H2）

**奥村:** 質問（1〜2文）

**実践者名:** 回答（1〜2段落、具体例を含む）

**奥村:** 要約コメント（1文）
```

**セクション数：** テーマに応じて 3〜6本。

**最終セクション** は以下の形で締める。見出しも他のセクション同様にキャッチコピー型にする（「最後に：〜」の定型は使わない）。

```markdown
---

## （キャッチコピー型の見出し）

**奥村:** 最後に、〜へ、ひと言いただけますか。

**実践者名:** キーメッセージを含む回答。

**奥村:** 今日はありがとうございました。

**実践者名:** こちらこそ、ありがとうございました。

---
```

### ブロックC：まとめ（アクションガイド）

```markdown
## 行動マップ：〇〇（テーマに応じたキャッチコピー型タイトル）

### カテゴリ名1
理念や原則を1文で。

### カテゴリ名2
- **1-1 ラベル：** 説明文。
- **1-2 ラベル：** 説明文。
- **1-3 ラベル：** 説明文。

### カテゴリ名3
- **2-1 ラベル：** 説明文。
- **2-2 ラベル：** 説明文。

### カテゴリ名4（締めの理念）
結びの1文。

---

構成・文／奥村 龍晃（日本キャッシュフローコーチ協会会員／キャッシュフローコーチ養成塾東京9期）
```

**まとめの設計ルール：**
- ナンバリングは「大分類-連番」形式（例：1-1, 1-2, 2-1）
- 箇条書きの各項目は `- **番号 ラベル：** 説明文。` の形式
- ラベルは省略可（ただし番号は必須）
- 最初のカテゴリは理念・原則（1文のみ、箇条書きなし）
- 最後のカテゴリは締めの理念（1文のみ、箇条書きなし）
- 中間のカテゴリはアクション項目（箇条書き）

## 対話パートの設計原則

### ストーリーライン

対話の流れは以下の5段階を基本とする。テーマに応じて変形・省略可。

1. **誤解の解消** — 読者が持ちそうな思い込みを崩す
2. **ニーズの証明** — 実践者のアプローチが求められる背景を示す
3. **具体的手法** — 再現可能な方法を提示する
4. **実行のコツ** — 実際に動くときの具体的ノウハウを伝える
5. **成果への接続** — 行動の先にある成果・循環を示す

### 奥村の質問パターン

文字起こしの内容に応じて、以下の型を使い分ける。

| 型 | 例 |
|------|------|
| 反論提示型 | 「〜という声があるが、どうお考えですか？」 |
| 疑問提示型 | 「〜でも、ちゃんと刺さるんでしょうか？」 |
| 直球質問型 | 「実際に〜は、どう〜するんですか？」 |
| 懸念提示型 | 「〜は大変じゃないですか？」 |
| 展開質問型 | 「〜の後は、どう〜につながっていくんでしょうか？」 |

### 奥村の要約コメント

- 各セクション末尾に1文で配置
- 実践者の回答を短く言い換え、読者の理解を確認するチェックポイントとして機能させる
- 語尾は「〜ですね」「〜わけですね」で統一

### 見出しの設計

見出しには3つのレベルがある。Lv.3「読者巻き込み型」を目指す。

| レベル | 特徴 | 判定基準 |
|---|---|---|
| Lv.1 要約型 | 内容をそのまま短くしただけ | ×使わない |
| Lv.2 報告型 | 「実践者が何をしたか」のキャッチコピー | △まだ弱い |
| Lv.3 読者巻き込み型 | 読者の声・具体シーン・数字で引き込む | ○目指す |

**Lv.3にするための3つの手法**

1. **読者の声を入れる：** 読者の頭の中にある悩み・疑問・思い込みを鉤括弧で見出しに入れる。読者が「自分の話だ」と感じて引き込まれる
   - 例：`「顧問契約が取れない」なら、研修から攻めればいい`
   - 例：`「単価が安い？」それでも研修会社を選んだ理由`
2. **本文中の生のセリフ・数字を引っ張り上げる：** 一番ドラマチックなシーンや具体的な数字を見出しに使う。「フィクションではない」信頼感と臨場感が出る
   - 例：`10分のブロックパズルで、その場でオファーが来た`
   - 例：`同じ手で10社にエントリー。半分が書類を通過した`
3. **見出し同士の構造を変える：** 全セクションが同じ長さ・同じトーンだと読者が飽きる。読者の声（鉤括弧）、短い断言、数字入り、セリフ引用と構造を変えてリズムに変化をつける

**文体・長さ**

- 動詞は過去形（「〜した」「〜狙った」）か体言止め（「〜獲得」「〜不要に」）で終わらせる
- 動詞の原形（「〜する」「〜できる」「〜入れる」）で終わらせない → 説明調になるため

**使えるパターン**

| パターン | 例 |
|---|---|
| 読者の声+提案型 | `「顧問契約が取れない」なら、研修から攻めればいい` |
| 読者の疑問+回答型 | `「単価が安い？」それでも研修会社を選んだ理由` |
| 断言2連打型 | `ライバル不在。財務・会計は穴場だった` |
| 数字+事実型 | `同じ手で10社にエントリー。半分が書類を通過した` |
| 手段+ドラマ結果型 | `10分のブロックパズルで、その場でオファーが来た` |
| メッセージ型（最終節向け） | `入口は財務。信頼をつかめば道は広がる` |

**禁止パターン**

- 動詞原形終わり：`〜で迂回する`、`〜を突破する`、`〜でも入れる`
- 「だから〜」型：`〜だから未経験でも入れる`（説明的すぎる）
- 見出しだけで内容が全部わかるもの（例：`模擬登壇は〜のブロックパズルで突破する`）
- 汎用的すぎて何の記事にでも使えるもの（例：`完璧な準備より、まず一歩`）
- 本文に書いていない内容を見出しに書くこと（見出しと本文の整合性を必ず確認する）

**見出しの最終チェック**

見出しを書いたら以下を確認する。
1. 見出しだけ並べて読んだ時に「ストーリー」が浮かぶか
2. 各見出しの構造（鉤括弧・数字・断言など）にバリエーションがあるか
3. 見出しの内容と本文の核心が一致しているか（本文の一番おいしい部分を拾えているか）
4. 読者が「自分事」として読める表現になっているか

## 文体ルール

### 対話パート（敬体：です・ます調）

**奥村：**
- 丁寧語
- 質問は1〜2文で簡潔に
- 要約コメントは共感・確認の語尾（「〜ですね」「〜わけですね」）
- 「なるほど」「確かに」など相槌を自然に入れる

**実践者：**
- 敬体だが口語寄り（「〜ですよ」「〜ですから」「〜ですね」）
- 具体的な台詞やシーンの描写を含める（「私はこういうお話しをします。『〜』と。」）
- 1つの回答の中に「主張 → 根拠 → 具体例」の流れを持たせる
- 自身の定番フレーズへの言及（「私がいつもお伝えしているのは」など）

### まとめパート（常体：だ・である調）

- 動詞の推奨形を使う（「始める」「入れる」「当てる」「出す」）
- 対話パートの内容を行動ステップとして再構成
- 簡潔に（各項目1〜2文）


## Markdown記法ルール

| 記法 | 用途 | 注意 |
|------|------|------|
| `# H1` | 記事タイトル | 1箇所のみ |
| `## H2` | セクション見出し | 対話セクション + まとめ |
| `### H3` | まとめ内サブ見出し | まとめパート内のみ |
| `**太字**` | 話者名ラベル、箇条書きラベル | `**奥村:**`、`**1-1 ラベル：**` |
| `---` | セクション区切り | 各対話セクションの前後 |
| `- ` | 箇条書き | まとめパート内のみ |
| `『』` | 書籍名 | 全角二重鉤括弧で統一 |
| `「」` | 発話の引用 | 全角鉤括弧で統一 |
| `""` | 言い回し・専門用語の強調 | 全角ダブルクォートで統一 |

**禁止事項：**
- 対話パート内での箇条書き使用（対話の自然さを損なう）
- H3 を対話パート内で使用（まとめパート専用）
- 太字の多用（話者名ラベルと箇条書きラベルのみ）

## ワークフロー

```
Task Progress:
- [ ] Step 1: 文字起こしからテーマとキーメッセージを抽出
- [ ] Step 2: 実践者プロフィール・経歴を確認（不足なら質問）
- [ ] Step 3: ストーリーラインを設計（5段階に当てはめる）
- [ ] Step 4: 導入部（ブロックA）を執筆
- [ ] Step 5: 対話パート（ブロックB）を構成・執筆
- [ ] Step 6: まとめパート（ブロックC）を構成・執筆
- [ ] Step 7: 全体の整合性・文体の統一を確認
```

**Step 1：テーマとキーメッセージの抽出**
文字起こしを読み、以下を特定する。
- 中心テーマ（何を・どう実践したか）
- キーメッセージ（実践者が最も伝えたいこと）
- 読者が持ちそうな誤解やハードル
- 具体的エピソード・台詞

**Step 2：プロフィール確認**
入力情報に実践者の氏名・期・肩書・経歴が揃っているか確認。不足があればユーザーに質問する。

**Step 3：ストーリーライン設計**
文字起こしの内容を5段階パターンに当てはめ、セクション構成を決める。文字起こしの順序にこだわらず、読者にとって最も分かりやすい流れに再構成する。

**Step 4〜6：執筆**
各ブロックの構造ルールに従って執筆。文字起こしの発言をそのまま使わず、読みやすい対話体に再構成する。ただし、実践者の個性的な言い回しや具体的エピソードは可能な限り活かす。

**Step 7：最終チェック**
- 対話パートとまとめパートで文体が混在していないか
- 見出しが要約型になっていないか（キャッチコピー型になっているか）
- 奥村の要約コメントが各セクション末にあるか
- Markdown記法が統一されているか
- まとめパートの番号体系が一貫しているか
